ASU · Graduate Applied Project · 2024–2025
Healthcare System Efficiency Evaluation Framework
Pythonscikit-learnSHAPK-MeansRandom ForestStreamlitDockerPanel RegressionWHO APIWorld Bank API
⌥ GitHub Repo ● Live Demo
The Problem

Healthcare systems worldwide face a fundamental question: are they getting enough health benefit per dollar spent? Traditional benchmarks compare countries by spending or outcomes in isolation. This project builds a composite efficiency metric — the Healthcare Efficiency Score (HES) — that captures the ratio of health output to spending, using objective entropy-based weighting so the data decides what matters.

Pipeline
01
Data Collection & CleaningRaw data from WHO, OECD, World Bank, Our World in Data — 4 sources, 52 countries, 2000–2022. APIs fetched programmatically via WHO GHED and World Bank APIs.
02
Feature Engineering & HES ScoringMin-max scaling of UHC coverage, financial protection, and CO2 intensity. Shannon entropy derives objective weights (CO2: 0.408, UHC: 0.351, Financial: 0.240). HES = weighted sum / log(1 + spending%).
03
Two-Way Fixed Effects RegressionPanel regression with country + year fixed effects isolates within-country causal impact of spending changes. Clustered standard errors by country for serial correlation.
04
K-Means Clustering (k=5)k selected via silhouette score maximization. Countries grouped into 5 archetypes by spending, coverage, and financial hardship patterns. Archetype labels inferred from cluster centroids.
05
Random Forest + SHAPTwo RF models (UHC target, financial hardship target). GroupKFold CV by country prevents leakage across train/test sets. SHAP waterfall plots show per-country variable contributions.
06
Streamlit Dashboard + Docker6-tab interactive dashboard with choropleth maps, cluster scatter plots, regression results, SHAP plots, US deep-dive, and a live policy simulator using FE regression marginal effects.
Key Results
#43 / 52US efficiency rank despite highest spending (16.5% of GDP)
k = 5Optimal archetypes via silhouette score — from High-Spend/Low-Coverage to Low-Spend/High-Efficiency
0.408Shannon entropy weight for CO2 intensity — highest discriminating variable across countries
52 countriesMega panel covering 2000–2022, merged from 4 international sources
Top Findings
The US Efficiency Paradox
US spends 16.5% of GDP (8.3pp above panel avg) yet ranks #43/52. UHC score of 87 is only 7 points above average despite nearly double the spending. The model isolates over-spending relative to outcomes as the structural driver.
Efficiency ≠ Spending
Turkey (HES 0.48) and Costa Rica (0.42) outperform all high-income spenders by achieving solid UHC coverage at 3–7% of GDP — proving efficient system design matters more than raw investment.
Shannon Entropy Weighting
Rather than assigning arbitrary weights, the index uses entropy to reward variables with more cross-country variation. CO2 intensity received the highest weight because it varies most — making the scoring fully data-driven.
What I'd Improve

Add causal inference via synthetic control or DiD for policy interventions. Incorporate time-lagged features (spending effects on outcomes have multi-year delays). Extend to sub-national data for countries with large regional variation. Add uncertainty bands to the policy simulator projections.

ASU · NLP Exploration · 2024
Cross-Domain Tone Classification
BERTPyTorchHuggingFaceTransfer LearningNLPPythonCalibration Mapping
⌥ GitHub Repo
Context
This is an exploratory research project, not a production system. Accuracy is modest on a genuinely hard cross-domain task. The value is in the problem framing and the design decision — learned calibration mapping — rather than benchmark-level results.

Real-world NLP systems often need to combine models trained on different datasets with incompatible label systems. You can't directly compare a "tone analysis" model (Formal/Informal) to a "GoEmotions" model (28 emotion categories) against a shared target (Polite, Professional, Casual). This project explores whether a small calibration set can learn that mapping — rather than hardcoding it by hand.

The Core Idea
01
Train separate models on incompatible label spacesModel B on Tone Analysis (2,684 samples), Model C on GoEmotions (8,044 samples). Each keeps its own label taxonomy — no manual unification at training time.
02
Learn the mapping from a calibration setA small split of the target dataset (Polite / Professional / Casual) is used to learn an optimal greedy mapping from each model's label space to the shared taxonomy. Data-driven, not handcrafted.
03
Evaluate on held-out test set4,800 samples. Confusion matrices and training curves generated. Results show Model B outperforms Model C despite having 3x fewer training samples.
Results
ModelTraining DataAccuracyMacro F1
Model B — Tone Analysis2,684 samples54.58%0.5337
Model C — GoEmotions8,044 samples46.50%0.4635
Note: Accuracy on a 3-class problem with a random baseline of ~33%. These numbers reflect a genuinely hard cross-domain transfer task with greedy (non-optimal) label mapping and a synthetic ground truth dataset. Results are directionally meaningful, not benchmark-ready.
The Key Takeaway
Domain alignment beats dataset size
Model B used 3x fewer samples but won — because its label space was more semantically aligned to the target task. This is a practically important signal: when building NLP systems from multiple sources, dataset selection matters more than scale alone.
Calibration mapping as a design pattern
Treating label mapping as a learned function (rather than a hardcoded lookup) is a viable approach to cross-dataset integration. The greedy implementation here has known limitations, but the pattern itself — calibrate don't curate — has real production relevance.
What I'd Do Differently

Replace greedy mapping with optimal transport for principled label alignment. Add a zero-shot baseline (GPT-4 or a modern instruction-tuned model) to benchmark the calibration approach against. Use adapter-based fine-tuning (LoRA) instead of full BERT fine-tuning to reduce compute and overfitting risk. Generate a non-synthetic ground truth dataset to strengthen validity of results.

Team Computers · Wind Energy · 2021–2022
Predictive Maintenance & Wind Turbine Analytics
PythonMLTableauSQLIoT DataForecastingKPI Dashboards
The Problem

Wind turbine failures are costly and hard to predict. Unplanned downtime can run into hundreds of thousands of dollars per turbine per day. The challenge was building a system that could ingest real-time operational and weather data, detect early failure signals, and surface actionable alerts to maintenance teams — before the turbine went down.

What I Built
01
IoT Data PipelineIngested real-time sensor data (vibration, temperature, RPM, power output) alongside weather feeds (wind speed, direction, pressure). Built SQL ETL pipelines to clean, merge, and store at hourly granularity.
02
Predictive Maintenance ModelsTrained ML models on historical failure data to identify early-warning signatures. Engineered lag features, rolling statistics, and weather-operational interaction terms as predictors.
03
Energy ForecastingIntegrated weather forecast data with turbine performance baselines to produce production forecasts. Helped operations teams plan maintenance windows without sacrificing output.
04
KPI Dashboards (Tableau)Built real-time dashboards tracking turbine health scores, predicted failure probability, energy production vs forecast, and maintenance scheduling recommendations.
Outcomes
↓ 20%Reduction in unplanned turbine downtime
↓ 15%Reduction in maintenance costs via proactive scheduling
↑ 15%Improvement in energy production efficiency
Real-timeLive KPI dashboards across turbine fleet for operations team
Key Lesson
Weather context transforms performance models
Raw sensor data alone had weak predictive power. The biggest performance gains came from joining operational readings with weather variables — wind speed interactions with RPM created the most discriminating failure features. Domain knowledge about how physical variables interact was as important as model selection.
