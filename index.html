Personal Project · Agentic AI · LangGraph + RAG · 2025
Personal Trainer AI Agent
PythonLangGraphLangChainChromaDBRAGPubMed APISQLitesentence-transformers
⌥ GitHub Repo
The Problem

Generic fitness apps give the same advice to everyone and forget you between sessions. The goal was an agent that knows the member before the first message, grounds every recommendation in real research, and remembers the conversation across sessions rather than just within one.

Architecture
01
LangGraph state graphTrainerState holds messages, user_id, current_plan, and session_count. Two nodes: chatbot (Claude with bound tools) and tools (LangGraph ToolNode). Conditional edge on last message tool_calls routes between them. Loops chatbot to tools and back until no more tool calls, then END.
02
SQLite checkpointer for persistent memorySqliteSaver from langgraph.checkpoint.sqlite persists full conversation state keyed by thread_id, set to the member PT-XXXX ID. The agent picks up exactly where it left off across sessions.
03
RAG knowledge basePubMed abstracts fetched via Biopython Entrez across 8 fitness topics (hypertrophy, protein intake, progressive overload, rest intervals, compound movements, injury prevention, sleep recovery, nutrition timing) combined with curated exercise markdown. Chunked with RecursiveCharacterTextSplitter, embedded with all-MiniLM-L6-v2, stored in ChromaDB.
04
5 LangChain tools via @tool with member_id closurelog_member_workout, get_member_workout_history, create_workout_plan, get_member_profile, and search_fitness_knowledge. member_id is closed over in build_tools() so Claude never needs to pass it as an argument.
05
Profile injection at session startMember profile (age, weight, goals, injuries, training days) is injected into the system prompt template at login via load_system_prompt(). Claude has full context before the first message. get_member_profile exists only for edge cases like mid-session profile updates.
Design decisions worth noting
search_fitness_knowledge runs before any advice
The tool description says "use this before giving any fitness advice." That single instruction means Claude consistently retrieves relevant PubMed abstracts and exercise data before responding, grounding recommendations in actual research rather than training data alone.
SQLite over in-memory state
Most agent demos use in-memory conversation history that dies when the process ends. SqliteSaver persists the full LangGraph checkpoint by thread_id so the agent picks up exactly where it left off. This is the difference between a demo and something that could actually be used.
Two branches, two deliberately different designs
The main branch implements the agent loop from scratch using the raw Anthropic API with no frameworks. This branch uses LangGraph and LangChain. Both solve the same problem. The contrast was intentional: understand the mechanics first, then use the abstraction that fits the task.
What I would build next

A Streamlit front end to make it shareable beyond the CLI. Progress tracking across weeks with visualisations pulled from workout history. Expand the knowledge base to cover nutrition. Add parallel tool use to fetch history and search the knowledge base in one round trip.

ASU · Graduate Capstone · Jan–May 2026
Healthcare System Efficiency Evaluation Framework
Pythonscikit-learnSHAPK-MeansRandom ForestStreamlitDockerPanel Regression
⌥ GitHub Repo ● Live Demo
The Problem

Healthcare systems worldwide face a fundamental question: are they getting enough health benefit per dollar spent? This project builds a composite efficiency metric, the Healthcare Efficiency Score (HES), using objective entropy-based weighting so the data decides what matters rather than arbitrary assumptions.

Pipeline
01
Data Collection & CleaningWHO, OECD, World Bank, Our World in Data. 4 sources, 52 countries, 2000–2022. APIs fetched programmatically.
02
HES Scoring via Shannon EntropyMin-max scaling of UHC coverage, financial protection, and CO2 intensity. Entropy weights: CO2 (0.408), UHC (0.351), Financial (0.240). HES = weighted sum / log(1 + spending%).
03
Two-Way Fixed Effects RegressionCountry and year fixed effects isolate within-country causal impact of spending. Clustered standard errors by country.
04
K-Means Clustering (k=5)k selected via silhouette score. 5 country archetypes from High-Spend/Low-Coverage to Low-Spend/High-Efficiency.
05
Random Forest + SHAPGroupKFold CV by country prevents data leakage. SHAP waterfall plots show per-country variable contributions.
06
Streamlit Dashboard + Docker6-tab dashboard: choropleth maps, cluster plots, regression results, SHAP plots, US deep-dive, live policy simulator.
Key Results
#43 / 52US efficiency rank despite highest spending (16.5% of GDP)
k = 5Optimal country archetypes via silhouette score
0.408Shannon entropy weight for CO2, highest discriminating variable
52 countriesPanel data 2000–2022 from 4 international sources
Top Findings
The US Efficiency Paradox
US spends 16.5% of GDP yet ranks #43/52. UHC score of 87 is only 7 points above average despite nearly double the spending. The model isolates over-spending relative to outcomes as the structural driver.
Efficiency is not the same as spending
Turkey (HES 0.48) and Costa Rica (0.42) outperform all high-income spenders at 3–7% of GDP. Efficient system design beats raw investment.
Entropy-driven objectivity
Shannon entropy rewards variables with more cross-country variation. CO2 intensity varies most across countries, earning the highest weight. Fully data-driven with no arbitrary assumptions.
What I'd improve

Causal inference via synthetic control or DiD. Time-lagged spending features. Sub-national data extension. Uncertainty bands on the policy simulator.

Personal Project · Gen-AI · LangGraph · 2025
AI News Summarizer
PythonLangGraphLangChainClaude HaikuNewsAPIROUGE-LLLM-as-JudgeStreamlit
⌥ GitHub Repo ● Live Demo
The Problem

News articles are long, written for general audiences, and hard to evaluate at scale without human raters. The goal was a system that could fetch live articles on any topic, rank them by relevance, summarize the top ones with Claude, score output quality automatically, and if quality fell short, refine the query and try again without any human intervention.

Graph Architecture
START → scrape → relevance → summarize → evaluate → router
                                                  |
                      quality low? → refine → scrape (up to 3x)
                      quality OK? → END
01
scrapeFetches articles from NewsAPI and scrapes full text from each URL via BeautifulSoup. Passes raw article text into the graph state.
02
relevanceClaude scores each article against the search topic. Top 3 by relevance score are kept for summarisation. Low-signal articles are dropped before any expensive summarize call.
03
summarizeClaude Haiku summarises each top article in 2–4 sentences. Prompt-engineered for factual fidelity with no padding.
04
evaluateTwo LLM-as-Judge chains score faithfulness and conciseness independently. ROUGE-L measures content recall against the source. All three scores stored per article in graph state.
05
router (conditional edge)If average quality meets the threshold, exit. If below threshold and iterations are under 3, route to refine. Max iterations guard prevents infinite loops.
06
refineClaude rewrites the search query to be more specific before the next scrape iteration. The graph adapts the query rather than retrying blindly.
Key Findings
LLM-as-Judge and ROUGE-L are complementary, not redundant
ROUGE-L measures surface-level content recall and penalises paraphrasing even when meaning is preserved. LLM-as-Judge catches faithfulness violations and padding that ROUGE scores as fine. Running both catches failure modes that neither finds alone.
Relevance filtering is the highest-leverage step
Ranking articles before passing them to Claude meant the summariser was always working on the right material. Removing this filter degraded faithfulness scores significantly. The model was faithful to the wrong content.
Query refinement outperformed prompt tuning
When summaries scored low, rewriting the NewsAPI query produced a bigger quality jump than adjusting the summarisation prompt. Bad input was the bottleneck. The model was doing its job; the search was pulling weak articles.
What I'd improve

Swap ROUGE-L for BERTScore for semantic similarity beyond surface overlap. Add parallel summarisation across articles to cut latency. Extend the eval rubric to cover source attribution. Add streaming output in the Streamlit UI.

ASU · NLP Exploration · 2024
Cross-Domain Tone Classification
BERTPyTorchHuggingFaceTransfer LearningNLPPython
⌥ GitHub Repo
Context
Exploratory research project, not a production system. The value is in the problem framing and the design decision around learned calibration mapping rather than benchmark-level results.

Real-world AI systems often need to combine models trained on incompatible label systems. This project explores whether a small calibration set can learn that mapping automatically rather than hardcoding it by hand.

The Core Idea
01
Train on incompatible label spacesModel B: Tone Analysis (2,684 samples). Model C: GoEmotions (8,044 samples). Each keeps its own taxonomy with no manual unification.
02
Learn the mapping via calibration setA small split of the target dataset (Polite / Professional / Casual) learns the optimal greedy mapping. Data-driven rather than handcrafted.
03
Evaluate on held-out test set4,800 samples. Model B outperforms Model C despite having 3x fewer training samples.
Results
ModelTraining DataAccuracyMacro F1
Model B, Tone Analysis2,684 samples54.58%0.5337
Model C, GoEmotions8,044 samples46.50%0.4635
Note: Random baseline is approximately 33% on this 3-class problem. Greedy label mapping and synthetic ground truth are known limitations.
Key Takeaway
Domain alignment beats dataset size
3x fewer samples and better results because label space alignment mattered more than scale. A practically important signal for anyone building AI systems from multiple data sources.
Calibration mapping as a design pattern
Treating label mapping as a learned function rather than a hardcoded lookup is a viable approach to cross-dataset AI integration with real production relevance.
What I'd do differently

Replace greedy mapping with optimal transport. Add a zero-shot LLM baseline. Use LoRA instead of full fine-tuning. Generate a non-synthetic ground truth dataset.
