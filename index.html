ASU · Graduate Applied Project · 2024–2025
Healthcare System Efficiency Evaluation Framework
Pythonscikit-learnSHAPK-MeansRandom ForestStreamlitDockerPanel Regression
⌥ GitHub Repo ● Live Demo
The Problem

Healthcare systems worldwide face a fundamental question: are they getting enough health benefit per dollar spent? This project builds a composite efficiency metric — the Healthcare Efficiency Score (HES) — using objective entropy-based weighting so the data decides what matters, not arbitrary assumptions.

Pipeline
01
Data Collection & CleaningWHO, OECD, World Bank, Our World in Data — 4 sources, 52 countries, 2000–2022. APIs fetched programmatically.
02
HES Scoring via Shannon EntropyMin-max scaling of UHC coverage, financial protection, CO2 intensity. Entropy weights: CO2 (0.408), UHC (0.351), Financial (0.240). HES = weighted sum / log(1 + spending%).
03
Two-Way Fixed Effects RegressionCountry + year fixed effects isolate within-country causal impact of spending. Clustered standard errors by country.
04
K-Means Clustering (k=5)k selected via silhouette score. 5 country archetypes from High-Spend/Low-Coverage to Low-Spend/High-Efficiency.
05
Random Forest + SHAPGroupKFold CV by country prevents data leakage. SHAP waterfall plots show per-country variable contributions.
06
Streamlit Dashboard + Docker6-tab dashboard: choropleth maps, cluster plots, regression results, SHAP plots, US deep-dive, live policy simulator.
Key Results
#43 / 52US efficiency rank despite highest spending (16.5% of GDP)
k = 5Optimal archetypes via silhouette score
0.408Shannon entropy weight for CO2 — highest discriminating variable
52 countriesPanel data 2000–2022 from 4 international sources
Top Findings
The US Efficiency Paradox
US spends 16.5% of GDP yet ranks #43/52. UHC score of 87 is only 7 points above average despite nearly double the spending. The AI model isolates over-spending relative to outcomes as the structural driver.
Efficiency ≠ Spending
Turkey (HES 0.48) and Costa Rica (0.42) outperform all high-income spenders at 3–7% of GDP. Efficient system design beats raw investment.
Entropy-Driven Objectivity
Shannon entropy rewards variables with more cross-country variation — CO2 intensity varies most, earning the highest weight. Fully data-driven, no arbitrary assumptions.
What I'd Improve

Causal inference via synthetic control or DiD. Time-lagged spending features. Sub-national data extension. Uncertainty bands on the policy simulator.

Team Computers · Wind Energy · 2021–2022
Predictive Maintenance & Wind Turbine Analytics
PythonMLTableauSQLIoT DataForecasting
The Problem

Wind turbine failures are costly and hard to predict. Unplanned downtime can run into hundreds of thousands of dollars per turbine per day. The challenge was building an AI system that ingests real-time operational and weather data, detects early failure signals, and surfaces actionable alerts — before the turbine goes down.

What I Built
01
IoT Data PipelineReal-time sensor data (vibration, temperature, RPM, power output) + weather feeds. SQL ETL pipelines cleaned, merged, and stored at hourly granularity.
02
Predictive AI ModelsTrained on historical failure data. Engineered lag features, rolling statistics, and weather-operational interaction terms as predictors.
03
Energy ForecastingWeather forecasts + turbine performance baselines → production forecasts. Helped operations plan maintenance windows without sacrificing output.
04
Real-Time KPI DashboardsTableau dashboards tracking turbine health scores, failure probability, production vs forecast, and maintenance scheduling.
Outcomes
↓ 20%Reduction in unplanned turbine downtime
↓ 15%Reduction in maintenance costs
↑ 15%Improvement in energy production efficiency
Real-timeLive KPI dashboards across turbine fleet
Key Lesson
Weather context transforms AI performance
Raw sensor data alone had weak predictive power. Joining operational readings with weather variables — wind speed interactions with RPM — created the most discriminating failure features. Domain knowledge mattered as much as model selection.
ASU · IFT 533 Data Visualization · 2025
LendingClub Credit Risk Analytics Dashboard
TableauTableau Prep BuilderCredit RiskEDAChoropleth MapsData Visualization
◈ Live Tableau Dashboard ⌥ Data Source (Kaggle)
The Problem

Credit risk analysts need to quickly identify which borrower segments carry the highest default risk — and why. With 14.5M LendingClub loan records spanning 2007–2018, this project builds an interactive 3-tab Tableau dashboard that lets analysts explore default patterns, rejection drivers, and geographic risk without writing a single query.

3-Phase Approach
01
Phase 1 — PlanningDataset selection (14.5M records, 322 columns). Defined user personas, 12 analytical questions, and full data type inventory.
02
Phase 2 — Decision MakingTableau Prep Builder preprocessing: merged accepted/rejected datasets, derived default_flag, income_bucket, dti_bucket fields, outlier removal, sampling for Tableau Public size limits.
03
Phase 3 — Dashboard Implementation3-tab interactive dashboard with cross-filtering, tooltips, parameter-driven highlighting. Landing page KPIs + Accepted + Rejected application dashboards.
Dashboard Structure
◈
Landing Page14.5M total · 1.35M accepted · 13.2M rejected · Navigation buttons · 10 analysis questions overview
◈
Accepted Application DashboardDefault rate by loan purpose, grade/sub-grade, verification status · US state choropleth · DTI vs default scatter plot
◈
Rejected Application DashboardCredit score distribution · Top rejection purposes · Top 5 states by rejections · DTI ratio distribution
Key Results
14.5MTotal loan records analyzed (2007–2018)
~20%Default rate — 268K of 1.34M accepted loans charged off
A → GDirect correlation between loan grade and default rate
10Analytical questions answered across both dashboards
Key Findings
Grade is the strongest default predictor
Default rates increase monotonically from Grade A to Grade G — confirming LendingClub's grading as a reliable risk signal. Analysts can use grade filters to instantly segment high-risk portfolios.
Loan purpose matters more than expected
Car loans, credit card consolidation, and small business loans show the highest default rates. Wedding and vacation loans — despite seeming frivolous — have the lowest. Purpose is a stronger signal than income alone.
High DTI = wider default spread
Charged-off loans show significantly wider DTI distribution than fully paid ones. The scatter plot makes this relationship immediately visible — no statistical testing required.
ASU · NLP Exploration · 2024
Cross-Domain Tone Classification
BERTPyTorchHuggingFaceTransfer LearningNLPPython
⌥ GitHub Repo
Context
Exploratory research project, not a production system. The value is in the problem framing and the design decision — learned calibration mapping — rather than benchmark-level results.

Real-world AI systems often need to combine models trained on incompatible label systems. This project explores whether a small calibration set can learn that mapping automatically — rather than hardcoding it by hand.

The Core Idea
01
Train on incompatible label spacesModel B: Tone Analysis (2,684 samples). Model C: GoEmotions (8,044 samples). Each keeps its own taxonomy — no manual unification.
02
Learn the mapping via calibration setA small split of the target dataset (Polite / Professional / Casual) learns the optimal greedy mapping. Data-driven, not handcrafted.
03
Evaluate on held-out test4,800 samples. Model B outperforms Model C despite 3x fewer training samples.
Results
ModelTraining DataAccuracyMacro F1
Model B — Tone Analysis2,684 samples54.58%0.5337
Model C — GoEmotions8,044 samples46.50%0.4635
Note: Random baseline ~33% on this 3-class problem. Greedy label mapping and synthetic ground truth are known limitations.
Key Takeaway
Domain alignment beats dataset size
3x fewer samples, better results — because label space alignment mattered more than scale. A practically important signal for anyone building AI systems from multiple data sources.
Calibration mapping as a design pattern
Treating label mapping as a learned function rather than a hardcoded lookup is a viable approach to cross-dataset AI integration with real production relevance.
What I'd Do Differently

Replace greedy mapping with optimal transport. Add a zero-shot LLM baseline. Use LoRA instead of full fine-tuning. Generate a non-synthetic ground truth dataset.
